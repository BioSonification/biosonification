# Научная Архитектура И Методология

Этот документ описывает текущую постановку проекта после перехода на structured `v2` пайплайн. Основная задача теперь формулируется не как “найти истинную музыку внутри генома”, а как условная генерация музыки в стиле обучающего корпуса под управлением биологических признаков.

## 1. Постановка задачи

Вход:

- DNA, RNA или protein sequence

Выход:

- двухдорожечный символический MIDI:
  - гармония по тактам
  - монофоническая мелодия поверх этой гармонии

Цель:

- музыка должна быть похожа на обучающий корпус;
- биосигнал должен сдвигать гармоническое и мелодическое поведение;
- мелодия должна оставаться структурированной линией внутри гармонической формы.

## 2. Почему архитектура стала иерархической

Плоский генератор музыкальных событий вынужден одновременно решать три разные задачи:

- выбрать аккорд;
- выбрать мелодическую ноту;
- организовать вертикаль и форму.

На небольшом корпусе и компактной модели это быстро даёт хаос. Поэтому текущий пайплайн разделяет задачу:

1. `Bio -> Harmony`
2. `Bio + Harmony -> Melody`

Такой разрез лучше соответствует реальной музыкальной структуре и уменьшает пространство ошибок.

## 3. Биологический слой

Источник:

- `bio_music_pipeline/v2/bio.py`

### 3.1 Поддерживаемые типы последовательностей

- `dna`
- `rna`
- `protein`

### 3.2 Фрагментация

Длинные последовательности режутся на фрагменты:

- `fragment_length`
- `fragment_stride`
- `max_fragments_per_record`

Это нужно, чтобы:

- увеличить число train samples;
- не перегружать память;
- сделать pairing более стабильным.

### 3.3 Извлекаемые биологические признаки

Нуклеотидный блок:

- composition frequencies
- entropy
- GC content
- GC/AT skew
- codon entropy
- periodicity
- k-mer statistics

Белковый блок:

- перевод через longest ORF
- `Biopython ProtParam`
- aromaticity
- instability index
- isoelectric point
- GRAVY
- molecular weight
- flexibility mean/std
- secondary structure fractions
- amino acid composition

RNA structural block:

- `ViennaRNA`
- minimum free energy
- paired fraction
- loop fraction
- structural transition rate
- structural entropy

Опциональный белковый embedding:

- `transformers`
- `ESM`

### 3.4 Что получает музыкальная часть

Энкодер возвращает:

- `vector`: фиксированный `bio embedding` размерности `256`
- `control_profile`: 6-мерный нормализованный профиль
- `tonic_pc_hint`: подсказка по тональному центру

`control_profile` интерпретируется как:

- tempo tendency
- density
- harmonic change tendency
- register
- harmony complexity
- mode tendency

## 4. Музыкальный слой

Источник:

- `bio_music_pipeline/v2/structured_music.py`

### 4.1 Репрезентация

Музыкальный сегмент больше не хранится как единый event-stream. Он раскладывается на:

- `HarmonyBar`
- `MelodyEvent`
- `StructuredMusicSegment`

### 4.2 Извлечение гармонии

Для каждого окна по тактам:

- определяется локальный key estimate
- выполняется `chordify`
- на каждый такт извлекается root и quality
- если гармония не меняется, используется `hold`

Гармония задаётся по фиксированной сетке тактов. Это важное ограничение: модель не может хаотично переключать аккорды внутри такта.

### 4.3 Извлечение мелодии

Мелодическая партия выбирается либо по имени части:

- `melody`
- `soprano`
- `lead`
- `voice`

либо как партия с наибольшей средней высотой.

Далее каждая нота кодируется как:

- `bar_index`
- `position_step`
- `duration_steps`
- `relative_pc` относительно текущего аккорда
- `octave`

### 4.4 Нормализация мелодии

При декодировании включено жёсткое ограничение:

- мелодия не выходит за пределы гармонической сетки;
- события с одинаковым onset схлопываются в одну ноту;
- длительности обрезаются так, чтобы мелодия оставалась монофонической.

Это сознательное архитектурное решение: текущий этап должен строить именно одну ведущую мелодическую линию поверх аккордов.

## 5. Pairing

Источник:

- `bio_music_pipeline/v2/structured_pairing.py`

### 5.1 Что именно сопоставляется

Сопоставляются:

- bio fragments
- structured music segments

через пространство музыкальных дескрипторов.

### 5.2 Дескрипторы музыкального сегмента

Используются:

- tempo
- note density
- harmonic change rate
- register
- harmony complexity
- mode

### 5.3 Калибровка

Сначала bio-profile калибруется под распределение музыкального корпуса:

- `bio_mean`, `bio_std`
- `music_mean`, `music_std`

Это нужно, чтобы биологический контроль попадал в тот диапазон, где реально живёт обучающий корпус.

### 5.4 Matching

Далее строится `top-k` pairing с весами. Это many-to-many схема, а не жёсткая 1:1 привязка.

## 6. Генеративные модели

Источник:

- `bio_music_pipeline/v2/structured_model.py`

Используется один и тот же класс `BioConditionedSequenceModel`, но он обучается отдельно для двух задач.

### 6.1 Harmony model

Вход:

- `bio vector`
- control tokens

Выход:

- последовательность баров `BAR -> ROOT/QUAL` или `BAR -> HOLD`

### 6.2 Melody model

Вход:

- `bio vector`
- control tokens
- harmony prefix tokens

Выход:

- последовательность `BAR -> POS -> RELPC -> OCT -> DUR`

Такая факторизация даёт важный эффект: мелодия всегда генерируется уже внутри заданного гармонического контекста.

## 7. Обучение

Источник:

- `bio_music_pipeline/v2/structured_train.py`

Контур обучения:

1. кодирование FASTA
2. загрузка structured music corpus
3. раздельный split bio/music
4. pairing train/val/test
5. обучение `harmony model`
6. обучение `melody model`
7. test evaluation
8. smoke generation

Сохраняемые артефакты:

- `harmony_best.pt`
- `melody_best.pt`
- `structured_pipeline.pt`
- `metrics.json`
- `structured_sample.mid`

## 8. Генерация

Источник:

- `bio_music_pipeline/v2/structured_generate.py`

Контур инференса:

1. кодируется FASTA
2. применяется train-time calibration
3. генерируется `harmony`
4. гармония подаётся как prefix во вторую модель
5. генерируется `melody`
6. обе дорожки рендерятся в MIDI

## 9. Почему это уменьшает хаос

Текущая схема уменьшает хаос по четырём причинам:

- аккорды меняются только по тактам;
- мелодия выражена относительно текущего аккорда;
- токены мелодии декодируются под заданную гармонию;
- пост-обработка удерживает мелодию в одной линии и внутри формы.

## 10. Что проверено локально

На текущем устройстве с `RTX 2060 6 GB` подтверждено:

- `torch` работает на `cuda`
- тесты проходят
- обучение проходит end-to-end
- `harmony_loss` и `melody_loss` убывают
- smoke и final generation создают двухдорожечный MIDI фиксированной длины
- мелодия остаётся монофонической

## 11. Что пока не реализовано

Пока отсутствует третий этап:

- `Bio + Harmony + Melody -> Accompaniment`

То есть текущий результат уже строит гармонию и ведущую мелодию, но ещё не добавляет полноценный полифонический аккомпанемент поверх готовой линии.
